# DAW single-solo behavior

- Restricted DAW soloing so only one track can be soloed at a time.
- Updated the demo client to replace any existing solo selection instead of accumulating multiple solo tracks.
- Hardened the playback engine so an incoming project snapshot or live solo change normalizes to a single active solo track.
- Added regression coverage for the engine-level solo constraint and the UI interaction flow that swaps solo selection between tracks.

# 2026-07-15

## DAW track identity after rename

- Fixed the rename-then-add-track regression where duplicate placeholder
  cleanup could collapse distinct tracks by treating mutable `trackName` as an
  identity key.
- Changed client and server blank-track cleanup to reconcile only matching
  `trackId` values. Same-name tracks with distinct IDs are now preserved.
- Updated default `Track N` naming to consider stable lane position as well as
  generated labels, so renaming the first lane no longer makes the next add
  reuse `Track 1`.
- Added reducer coverage that replays rename, branch creation, and track
  creation, then verifies the renamed ID remains and the new ID is distinct.
- Strengthened rename, client/server cleanup, naming, and upload branch-builder
  tests to cover the identity invariant.
- Recorded the verified cause, corrective behavior, affected source, and
  regression evidence in
  [Session Problems and Solutions](../60_BRANCHES/daw-realtime-collab/session-problems-and-solutions.md#6-renaming-a-track-could-make-a-later-track-replace-it).
- Follow-up reproduction showed the identity cleanup corrected the new label to
  `Track 2` but did not stop replacement. Read-only database inspection found a
  populated branch followed by an empty automatic checkpoint, then a second
  branch containing only the newly added track.
- Fixed `createAutoDemoVersion` so automatic and semantic checkpoints clone the
  active source tracks instead of creating zero-track nodes. Logical `trackId`
  values survive the clone while immutable `trackVersionId` values remain new.
- Automatic checkpoint nodes are now recorded and broadcast as durable
  `VERSION_NODE_ADDED` operations with their parent and cloned track payload,
  preventing snapshot-tail bootstrap from rendering the next branch without
  its parent.
- Added regressions for the populated checkpoint-to-next-branch chain and the
  durable connected-node operation payload.
- Added current-bootstrap compatibility reconciliation for automatic nodes that
  were already persisted by the older event-only implementation but absent from
  snapshot operation tails. Historical sequence reads remain unchanged.
- Added the reusable
  [DAW Version-State Debugging Protocol](../01_PROTOCOLS/daw-version-state-debugging.md),
  covering identifier invariants, read-only database timelines, layered replay
  tracing, regression design, confidence calibration, and documentation handoff.
