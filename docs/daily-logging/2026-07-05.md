## Implementation Log

- Added a browser-level interaction regression test for the full `record on Track 1 -> stop/save -> add track` flow in `DemoDawClient`.
- Reproduced a real failure in that flow: the recording save path could throw while calling `scrollTo` on the tracks container in jsdom-like environments, which interrupted cleanup and left the temporary recording lane visible.
- Guarded the `scrollTo` call so recording save can finish cleanly even when the container does not implement that method.
- Kept the existing recording cleanup behavior intact so the committed track replaces the preview lane before the next add-track action.

## Duplicate Track Cleanup

- Added a shared duplicate-cleanup helper that prunes blank track versions when a track is added with the same `trackId` or same `trackName`.
- Wired the cleanup into the client reducer bootstrap/replay path so stale blank copies are removed as soon as project state is normalized or a `TRACK_VERSION_CREATED` operation lands.
- Wired the same cleanup into both server upload flows so add-track and recording uploads delete the blank duplicate server-side before the operation is committed.
- Added reducer and server helper regression tests that prove same-name audio tracks are preserved while blank duplicates are removed.
